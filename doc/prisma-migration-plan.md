# План внедрения PostgreSQL + Prisma в Knowledge Hub

## Цель

Перевести API с in-memory хранения на PostgreSQL через Prisma, не сломав текущие маршруты, форматы ответов и поведение существующих e2e-тестов.

## Что уже есть в проекте

- `docker-compose.yml` уже содержит сервис `db` на PostgreSQL и сервис `app`.
- Сейчас данные хранятся в массивах внутри сервисов:
  - `src/user/user.service.ts`
  - `src/article/article.service.ts`
  - `src/category/category.service.ts`
  - `src/comment/comment.service.ts`
- Логика каскадного удаления сейчас реализована вручную:
  - удаление пользователя обнуляет `authorId` у статей и удаляет его комментарии
  - удаление категории обнуляет `categoryId` у статей
  - удаление статьи удаляет связанные комментарии
- Фильтрация статей уже есть на уровне сервиса по `status`, `categoryId`, `tag`.

## Важное ограничение

Сначала переносим хранилище, а не переписываем API.

Это значит:

- URL не меняем
- DTO не меняем без необходимости
- статусы ответов должны остаться прежними
- форма JSON-ответов должна остаться прежней
- существующие e2e-тесты должны продолжить проходить

## Общая стратегия

Работу лучше делать в 10 шагах. После каждого шага желательно запускать минимальную проверку, чтобы не накапливать ошибки.

## Шаг 4. Описать схему Prisma

### Что сделать

Создать модели и enum'ы по заданию.

### Нужные enum'ы

- `UserRole`: `ADMIN`, `EDITOR`, `VIEWER`
- `ArticleStatus`: `DRAFT`, `PUBLISHED`, `ARCHIVED`

### Нужные модели

1. `User`
2. `Article`
3. `Category`
4. `Comment`
5. `Tag`

### Связи

- `User` -> `Article`: one-to-many через `authorId`
- `User` -> `Comment`: one-to-many через `authorId`
- `Category` -> `Article`: one-to-many через `categoryId`
- `Article` -> `Comment`: one-to-many через `articleId`
- `Article` -> `Tag`: many-to-many

### Правила удаления

- при удалении `User`:
  - `Article.authorId` -> `SetNull`
  - комментарии пользователя -> `Cascade`
- при удалении `Category`:
  - `Article.categoryId` -> `SetNull`
- при удалении `Article`:
  - комментарии статьи -> `Cascade`
  - связи с тегами удаляются автоматически

### Практическое решение по many-to-many

Можно использовать implicit many-to-many, если этого достаточно для задания. Это проще и быстрее.

Если нужен больший контроль, можно завести явную join-модель, но для текущей задачи это обычно избыточно.

### Важное проектное решение

Сейчас API отдает:

- `status` как lowercase строку: `draft`, `published`, `archived`
- `role` как lowercase строку: `admin`, `editor`, `viewer`
- timestamps как `number`
- `tags` как массив строк

Prisma по умолчанию будет работать с:

- enum-значениями в uppercase
- `DateTime` как `Date`
- many-to-many тегами как массив объектов

Значит, сразу надо запланировать слой маппинга Prisma -> API response, чтобы не сломать контракты.

## Шаг 8. Ввести слой маппинга Prisma -> текущий API

### Зачем это нужно

Сейчас доменные entity в проекте не совпадают с тем, что возвращает Prisma напрямую.

Примеры несовпадений:

- `Date` нужно превращать в timestamp number через `getTime()`
- `Tag[]` из Prisma нужно превращать в `string[]`
- enum из uppercase нужно превращать в lowercase
- из пользователя нельзя возвращать `password`

### Что лучше сделать

Создать отдельные mapper-функции, например:

- `toUserResponse(...)`
- `toArticleResponse(...)`
- `toCategoryResponse(...)`
- `toCommentResponse(...)`

### Почему это важно

Если вставлять маппинг прямо в каждый метод сервиса, код быстро станет хрупким и дублирующимся.

## Шаг 9. По очереди переписать сервисы с in-memory на Prisma

Лучше не переписывать все сразу. Надежнее идти по одному модулю.

### 9.1. CategoryService

#### Что заменить

- `findAll` -> `prisma.category.findMany()`
- `findOne` -> `prisma.category.findUnique()`
- `create` -> `prisma.category.create()`
- `update` -> `prisma.category.update()`
- `remove` -> `prisma.category.delete()`

#### Что проверить

- при удалении категории связанные статьи получают `categoryId = null`

### 9.2. UserService

#### Что заменить

- `findAll`
- `findOne`
- `create`
- `updatePassword`
- `remove`

#### Что проверить

- пароль не уходит в response
- при удалении пользователя статьи сохраняются, но `authorId` становится `null`
- комментарии пользователя удаляются

### 9.3. CommentService

#### Что заменить

- `findByArticleId`
- `findOne`
- `create`
- `remove`

#### Что проверить

- перед созданием комментария отсутствующая статья все еще дает `422 Unprocessable Entity`

Важно: Prisma по умолчанию при foreign key ошибке даст не тот же HTTP-ответ, что и сейчас. Значит, существование статьи нужно проверять явно до `create`.

### 9.4. ArticleService

#### Что заменить

- `findAll`
- `findOne`
- `exists`
- `create`
- `update`
- `remove`

#### Что особенно важно

Для тегов использовать `connectOrCreate` при `create` и `update`.

Примерная идея:

- если в DTO пришли `tags: ['nodejs', 'typescript']`
- для каждой строки выполнить `connectOrCreate`
- при update, если список тегов передан, сначала синхронизировать связи, а не просто дописывать новые

### Фильтрация статей

Метод `GET /article?status=...&categoryId=...&tag=...` должен перейти на Prisma `where`.

Примерная логика:

- `status` -> фильтр по enum
- `categoryId` -> фильтр по полю `categoryId`
- `tag` -> фильтр через relation `tags: { some: { name: tag } }`

### Важный нюанс

Поскольку API принимает lowercase статус, перед запросом в Prisma его нужно преобразовать в enum-значение Prisma.

## Отдельный список рисков

### 1. Несовпадение enum-значений

Сейчас в API используются lowercase-значения, а в Prisma enum обычно uppercase.

Значит, нужен явный конвертер в обе стороны.

### 2. Несовпадение формата дат

Сейчас `createdAt` и `updatedAt` возвращаются как `number`, а Prisma работает с `Date`.

Значит, response нужно маппить через `date.getTime()`.

### 3. Несовпадение структуры тегов

Сейчас `tags` это `string[]`, а Prisma вернет relation-объекты.

Значит, на выходе нужен `article.tags.map((tag) => tag.name)`.

### 4. Изменение статусов ошибок

Prisma-ошибки сами по себе не гарантируют те же HTTP-коды, что ожидают e2e.

Нужна ручная обработка случаев:

- `404 Not Found`
- `422 Unprocessable Entity`
- `403 Forbidden`

### 5. Проблемы с Docker hostname

Локальный запуск приложения и запуск в контейнере используют разный hostname для БД:

- локально: `localhost`
- в Docker: `db`

Это нужно учитывать сразу, иначе миграции будут падать.

## На что смотреть особенно внимательно

Этот блок полезно читать прямо во время реализации. Здесь собраны места, где новички чаще всего теряют совместимость или не до конца понимают, что именно происходит.

### 1. Граница между API-моделью и Prisma-моделью

Одна из самых важных мыслей в этой задаче: Prisma-модель и JSON-ответ API не обязаны совпадать один в один.

В этом проекте они как раз не совпадают:

- Prisma хранит `DateTime`, а API сейчас отдает `number`
- Prisma возвращает enum в своем виде, а API уже ожидает lowercase-строки
- Prisma по связи `tags` даст объекты, а API отдает массив строк

На это стоит смотреть особенно внимательно, потому что типовая ошибка новичка: вернуть данные из Prisma “как есть” и случайно сломать e2e.

### 2. Где должна жить бизнес-логика, а где инфраструктура

Prisma отвечает за доступ к данным, но не должен определять HTTP-поведение приложения.

Например:

- `404` возникает не потому, что Prisma так захотел, а потому что ваш сервис так решил обработать отсутствие записи
- `422` для комментария к несуществующей статье тоже нужно сохранить вручную
- `403` при неверном старом пароле тоже остается бизнес-правилом сервиса

То есть Prisma не заменяет сервисную логику, он заменяет только хранилище.

### 3. Разница между `undefined` и `null`

Это критично для update-операций.

- `undefined` обычно означает: поле не трогаем
- `null` означает: явно очищаем значение

В вашем проекте это особенно важно для:

- `authorId`
- `categoryId`
- `tags` при обновлении статьи

Если это перепутать, update начнет вести себя не так, как ожидает API.

### 4. Обновление many-to-many тегов

Создать теги через `connectOrCreate` обычно несложно. Сложнее корректно обновить список тегов статьи.

На что смотреть:

- если `tags` не пришли в DTO, старые связи должны сохраниться
- если `tags: []` пришли явно, связи должны очиститься
- если пришел новый массив, он должен заменить старое состояние, а не просто дописаться сверху

Это один из самых важных участков всей задачи.

### 5. Каскады на уровне БД против ручной логики в коде

Сейчас каскадное поведение реализовано вручную в сервисах. После перехода на Prisma и PostgreSQL значительная часть этого будет обеспечиваться foreign key constraints и `onDelete`.

Важно понимать:

- если правило можно надежно описать в схеме БД, лучше положиться на БД
- если нужно сохранить нестандартный HTTP-ответ или кастомную валидацию, это остается в коде

### 6. Не доверять только TypeScript-типам

Типы помогают, но не гарантируют корректное поведение.

Надо отдельно проверять:

- какие данные реально пришли из Prisma
- в каком виде ушел JSON-ответ
- проходит ли e2e на фактическом HTTP-контракте

### 7. Проверять поведение удаления через реальные сценарии

Важно проверить не только “запись удалилась”, но и все побочные эффекты:

- удалили пользователя -> статьи остались, `authorId` обнулился, комментарии исчезли
- удалили категорию -> статьи остались, `categoryId` стал `null`
- удалили статью -> комментарии исчезли, связи тегов удалились

## Мини-конспект для самопроверки перед защитой

Если сможете уверенно объяснить следующие тезисы, вы хорошо понимаете задачу:

- Prisma заменяет способ хранения данных, но не заменяет бизнес-логику сервиса
- схему БД нужно проектировать под правила предметной области, а не только под текущий код
- API response почти всегда требует отдельного слоя маппинга
- каскады и `SetNull` нужно осознанно выбирать по смыслу данных
- `undefined` и `null` в update-операциях означают разное
- many-to-many обновляется аккуратно, особенно если нужно заменить существующий набор связей
- успешная миграция это не только работающая база, но и сохраненная совместимость текущего API

## Definition of Done

Можно считать задачу завершенной, когда выполнены все пункты:

- есть `prisma/schema.prisma`
- есть миграции в `prisma/migrations/`
- есть `prisma/seed.ts`
- `npx prisma db seed` отрабатывает успешно
- приложение использует Prisma вместо in-memory массивов
- каскадные правила работают через БД
- фильтрация статей работает через Prisma `where`
- теги у статей создаются и обновляются через many-to-many
- `.env` не закоммичен
- `.env.example` присутствует
- текущие e2e-тесты проходят
